## Output Structure

Combined markdown with heading hierarchy for Neovim folding. Must begin with:

```
> **This review was generated by AI. Dave prolly read through it an cleaned it up but don't count on it.**
> Use your own judgment — findings may contain false positives or miss real issues.
```

Then two phases in order:

**Phase 1 — Comprehension** (per `code-review-comprehension` skill):
- `## Summary` — changeset TL;DR + behavioral diff
- `## Change Flow` — unified flow diagram with change markers + legend
- `## How It Works` — step-by-step walkthrough with `### Step N:` subheadings

**Phase 2 — Analysis** (per `code-review-rules` skill):
- `## Findings` — grouped by concern (`###`), individual findings (`####`)
- `## Positive Observations`
- `## Assessment`

## Local Changes

1. `git rev-parse --show-cdup` → cwd-to-root prefix.
2. `git diff` + `git diff --cached`. Both empty → `git diff HEAD~1`.
3. `git status`. Read source for context.
4. Generate both phases. Save. Chat: path + summary only.

## GitHub PRs

1. `gh pr view <url> --json title,body,baseRefName,headRefName,files,additions,deletions`.
2. `gh pr diff <url>`. Local reads for context if same repo.
3. Generate both phases. Save. Chat: path + summary only.

## Rules

- Read-only. Observe, analyze, report.
- NEVER modify source. Only `.dk-notes/reviews/` for reviews.
- NEVER output review in chat.
- Bash: git/gh read commands only.
- Off-topic? Refuse, explain read-only reviewer.
